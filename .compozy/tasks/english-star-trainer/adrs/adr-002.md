# ADR-002: Organizar o MVP como monorepo com React no frontend e NestJS no backend

## Status

Accepted

## Date

2026-05-24

## Context

O repositório ainda não possui aplicação implementada, então a arquitetura inicial precisa ser definida do zero. O produto exige uma interface web para criar sessões, responder perguntas com timer e consultar histórico, além de um backend para persistência em SQL Server local via Docker. O usuário pediu explicitamente React no frontend e NestJS no backend, mantidos no mesmo repositório, mas separados entre si.

## Decision

O MVP será organizado como um monorepo com duas aplicações principais: `frontend` em React para a experiência do usuário e `backend` em NestJS para API, regras de sessão e persistência. As aplicações permanecerão separadas por responsabilidade, mas compartilhando o mesmo repositório para simplificar desenvolvimento local e evolução coordenada do produto.

## Alternatives Considered

### Alternative 1: Monólito web em um único projeto full-stack

- **Description**: Usar um único projeto para renderização web, API e persistência.
- **Pros**: Menor número de aplicações para subir e coordenar.
- **Cons**: Conflita com a preferência explícita do usuário por React e NestJS separados.
- **Why rejected**: Não atende à stack desejada nem ao limite claro entre frontend e backend.

### Alternative 2: Frontend e backend em repositórios independentes

- **Description**: Criar um repositório para a interface e outro para a API.
- **Pros**: Isolamento total entre ciclos de entrega.
- **Cons**: Aumenta overhead operacional desnecessário para um MVP de uso individual.
- **Why rejected**: Adiciona complexidade de coordenação sem benefício proporcional no estágio atual.

## Consequences

### Positive

- Preserva separação clara de responsabilidades entre interface e API.
- Facilita evolução incremental do MVP mantendo baixo overhead de gestão.
- Permite modelar API e frontend de forma independente sem dispersar artefatos.

### Negative

- Exige orquestração local de mais de um processo de aplicação.
- Introduz integração HTTP entre frontend e backend desde o início.

### Risks

- Pode haver duplicação de contratos se os tipos compartilhados não forem bem definidos.
- Configuração local pode ficar frágil se scripts de inicialização não forem simples.

## Implementation Notes

O backend deve expor apenas a superfície mínima necessária para sessões, respostas e histórico. O frontend deve concentrar a experiência de treino e tratar o backend como fonte única de persistência.
