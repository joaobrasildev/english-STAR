# ADR-003: Persistir respostas do MVP em uma única tabela no SQL Server

## Status

Accepted

## Date

2026-05-24

## Context

O MVP precisa salvar pergunta, resposta completa consolidada e tempo total de resposta, além de permitir resumo da sessão atual e consulta de histórico simples. O usuário indicou SQL Server local em Docker como banco principal e não exigiu um modelo relacional mais rico, preferindo a opção mais simples quando perguntado sobre estrutura de dados. Também foi definido que a resposta só será salva quando o usuário confirmar o término da questão.

## Decision

O backend persistirá cada resposta finalizada em uma única tabela de SQL Server. Essa tabela incluirá identificador da resposta, identificador lógico da sessão, ordem da pergunta na sessão, texto da pergunta, resposta consolidada, tempo alvo em segundos, tempo total em segundos e timestamps mínimos para suporte ao histórico.

## Alternatives Considered

### Alternative 1: Separar sessões e respostas em tabelas distintas

- **Description**: Criar uma tabela para sessões e outra para respostas vinculadas.
- **Pros**: Modelo mais normalizado e mais flexível para relatórios futuros.
- **Cons**: Acrescenta joins, migrações e complexidade sem necessidade imediata no MVP.
- **Why rejected**: O produto atual precisa de histórico simples e o usuário priorizou a opção mais enxuta.

### Alternative 2: Persistência local sem banco relacional

- **Description**: Salvar dados em arquivo local ou armazenamento embutido apenas no ambiente de desenvolvimento.
- **Pros**: Menor esforço inicial de infraestrutura.
- **Cons**: Contraria o requisito explícito de usar SQL Server local em Docker e dificulta evolução do histórico.
- **Why rejected**: Não atende ao requisito de persistência escolhido para o MVP.

## Consequences

### Positive

- Mantém o modelo de dados simples e suficiente para o escopo aprovado.
- Facilita listagem de histórico por sessão com filtros básicos.
- Reduz o número de componentes e migrações iniciais.

### Negative

- Duplica dados de sessão em múltiplas linhas.
- Limita análises futuras mais sofisticadas sem evolução do schema.

### Risks

- Se o histórico crescer em complexidade, a tabela única pode ficar apertada para novas consultas.
- Regras de agrupamento por sessão precisarão ser consistentes para evitar ambiguidade na UI.

## Implementation Notes

Cada registro deve ser criado apenas ao confirmar o fim da questão. O agrupamento por sessão deve ser suficiente para montar o resumo final e a tela simples de histórico sem exigir entidades adicionais no MVP.
